# M2 — Academic Foundation — 🔧 IN PROGRESS

Scope (supersedes the earlier draft; per M2 kickoff brief 2026-07-05): **academic
structure only** — `AcademicYear`, `AcademicTerm`, `Class`, `Section`, `Subject`,
`TeacherAssignment`. Out: students/enrollment/attendance/marks/report cards/
timetable/fees, people records (Staff/Guardian), bulk import, class-teacher flag.
Naming note: this milestone uses **Class/Section** (older docs said ClassLevel/Division).

Roles: SUPER_ADMIN + OFFICE_ADMIN manage (`academic:manage`); staff read
(`academic:read`); teachers additionally read **their own** assignments via a
`ScopeRule`; parents have no access.

## Steps
- ✅ 1 — Requirements analysis (hierarchy, relationships, future deps, validation
  rules, lifecycles, edge cases, ordering rationale — delivered 2026-07-05).
  Key decisions: Class/Section/Subject are structural (not year-bound; enrollment
  binds time in M3); Subject is a school-wide catalog; `TeacherAssignment.teacherId
  → User.id` (no Staff table), role/status checked in business; assignments NOT
  year-scoped (unique `(teacher, subject, section)` per brief — year scoping is an
  additive M3+ migration if needed); no soft deletes (Restrict FKs + audit rows);
  one-ACTIVE-year via partial unique index; term overlap = inclusive-date
  intersection, enforced in business (DB EXCLUDE constraint decided at Step 2);
  every mutation audited in-transaction.
- ⬜ 2 — Database design (6 tables: PK/FK/constraints/indexes, audit-compatible)
- ⬜ 3 — RLS planning + policies (admin-manage / teacher-own-read / parent-none; documented)
- ⬜ 4 — Business layer (AcademicYear/AcademicTerm/Class/Section/Subject/TeacherAssignment services; validation, transactions, rules, audit; + `ConflictError` in core)
- ⬜ 5 — API layer (`academicYear.* academicTerm.* class.* section.* subject.* teacherAssignment.*` — Zod, thin routers)
- ⬜ 6 — Mobile (placeholder read-only screens: years, classes, subjects, assignments)
- ⬜ 7 — Web (full CRUD: search/pagination/filters/create/edit/delete-where-applicable)
- ⬜ 8 — Testing (unit/business/API/validation/authorization; edge cases: duplicates, multiple active years, overlapping terms, invalid dates) + ts/lint/tests/build
- ⬜ 9 — Documentation (API inventory, feature docs, project memory, status, milestone)
- ⬜ 10 — Deliverables report (completed, DB summary, architecture review, validation results, M3 remaining work, risks) → STOP for approval

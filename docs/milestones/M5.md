# M5 — Examination & Assessment — ✅ STEPS 1–10 COMPLETE (awaiting approval)

Scope (per M5 kickoff brief): **Exam, Assessment (Exam×Subject), ExamSection (the
register), Mark (result snapshot per Enrollment), GradeScale/GradeBand, central
grade compute, GPA foundation.** Out: report cards, promotion, homework,
timetable, fees, notifications (incl. publish-notify — brief overrides PRD),
analytics, exam attempts/re-exams — later milestones.

Roles: SUPER_ADMIN + OFFICE_ADMIN full management + publish; TEACHER enters +
reads marks for **own subject×section** and submits registers; PARENT reads
**published** marks for **own child** only; ACCOUNTANT none. **ADR-012 is the
source of truth** (extends ADR-011): marks key to **Enrollment, never Student**
(ADR-010); the register is an `ExamSection` (the `AttendanceSession` analog)
holding `Mark`s (the `AttendanceRecord` analog); ownership derives from
`TeacherAssignment` (never stored); lifecycle is forward-only `DRAFT → SUBMITTED
→ LOCKED`; grade/percentage is **snapshotted at lock**; GradeScale edits never
mutate history; **two grains** — lock per register, publish per exam.

## Steps

- ✅ 1 — Requirements analysis: exam/assessment/register/mark grain, the two
  axes (lock per ExamSection, publish per Exam), grade snapshots, GPA from
  snapshots, why marks belong to Enrollment, the cross-year wrinkle (R1), the
  publish-over-incomplete choice (R3), the cascade-vs-published risk (R5).
  Recorded in **ADR-012** (15 refinements folded in). Naming diverges from the
  PRD: `Assessment` (not `ExamSubject`), `Subject` directly (no `ClassSubject`),
  `ExamType` enum (not `category String`), `maxPractical` nullable.
- ✅ 2 — Database design (migration `20260709000000_examination_assessment`): 5
  models (`Exam`, `Assessment`, `ExamSection`, `Mark`, `GradeScale`/`GradeBand`)
  + `ExamType`/`ExamSectionStatus` enums. Invariants in the DB: uniques
  (`(assessmentId, sectionId)` register, `(assessmentId, enrollmentId)` mark, one
  default scale), CHECKs (mark limits), a `GradeBand` non-overlap `EXCLUDE`
  (btree_gist, `numrange(min,max,'[)')`), Restrict/Cascade/SetNull delete rules.
  Empirically verified against Postgres (drift-check + 15 seeded constraint
  proofs).
- ✅ 3 — Relationships: definition chain `Exam→Assessment→ExamSection→Mark` all
  `onDelete: Cascade`; actor FKs Restrict (audit actors durable); mark dual-FK to
  assessment + enrollment (transfer-safe natural key). All delete-rule edges
  proven. **R5 published-data deletion guard implemented early** (at user
  direction, before Step 4): `deletion.service.ts` rejects deleting a published
  exam or one with any LOCKED section (guarded, audited).
- ✅ 4 — RLS (migration `20260709010000_exam_rls`, dedicated): admin ALL; teacher
  own subject×section SELECT/INSERT/UPDATE (ExamSection/Mark), own-subject/own-
  exam SELECT; parent published-own-child Mark + published Exam/Assessment SELECT;
  GradeScale/GradeBand read-only reference; anon none; service_role bypass.
  SECURITY DEFINER helpers (teaches_subject/exam/section, exam_published_for_
  section requiring LOCKED). Isolation proven **12/12 read + 15/15 write** on
  local Postgres with stubbed `auth.uid()`.
- ✅ 5 — Business layer: 5 services under `services/exam/` (exam, assessment,
  mark, grade, gradeScale) + shared `scope.ts` (B3 staff-actor, derived
  ownership, parent visibility) + mappers. Central grade compute in
  `@repo/core/grade.ts` (pure; rounds %→2dp before band lookup; absent→null).
  `saveMarks` auto-creates the register **race-safe** (`ensure` upsert — a real
  race was found and hardened); lock computes + snapshots in one tx and rejects
  incomplete/scale-gap; unlock requires a reason. 4 new repos wired into
  `Repositories`.
- ✅ 6 — API layer: 4 thin routers (`exam/assessment/mark/gradeScale`, 21
  procedures) on `protectedProcedure`; Zod inputs in `@repo/validation`. No
  logic/Prisma/role strings in transport.
- ✅ 7 — Mobile: teacher — markable list (assessment×section discovery) → mark
  entry (theory/practical/absent, save-draft → submit; SUBMITTED/LOCKED read-
  only); parent — child picker → published results + GPA. **Gap filled:**
  `mark.markable` (teacher discovery — admin catalogs are gated) + DTO name
  enrichment (subject/exam names) so scope-limited clients can label rows.
- ✅ 8 — Web (admin console): `/exams/{,[examId],grade-scales}` gated on
  `exam:manage` — exam dashboard + CRUD + **publish** (R3 locked-vs-total
  confirm), assessment CRUD, register oversight + a **marks grid** (admin entry,
  submit/lock/unlock, CSV), grade-scale management. **Two admin reads added:**
  `exam.get`, `exam.registers` (`listExamRegisters` — admins have no
  `TeacherAssignment`, so this is their register-discovery path; also feeds the R3
  count). **R3 lives in the web** (publishExam is count-agnostic).
- ✅ 9 — Testing: business (`exam.services.test.ts` — exam CRUD/publish/

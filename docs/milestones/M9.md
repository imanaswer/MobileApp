# M9 — Timetable Management — ✅ STEPS 1–10 COMPLETE (awaiting approval)

Scope: a **new read-mostly domain** — period definitions, a per-year bell schedule, and a weekly
section timetable that yields teacher / section / parent / today views over frozen M1–M8. Purely
**additive** (three new tables, one enum, two adopted permission constants; zero frozen-table change).
Ownership derives **only** from `TeacherAssignment`; double-booking is structurally impossible; every
mutation is audited in-transaction. **Notifications are out of scope.** **ADR-017** governs the design.

## Deliverables

| Area | What shipped |
|---|---|
| Schema | `BellSchedule` (one/year) → `Period` (clock-time slots) → `TimetableEntry` (section×weekday×period → subject+teacher+room) + `enum Weekday` |
| Conflict rules | Section + teacher double-booking = DB uniques (friendly business pre-checks); period overlap / order clash / cross-year / no-class-on-break = business layer |
| Ownership | Derived from `TeacherAssignment(teacher, subject, section)` — **never** `ClassTeacherAssignment` |
| API | `bellSchedule.*`, `period.*`, `timetable.*` (14 procedures) — thin validate→delegate |
| Mobile | Read-only `/timetable` (teacher own / parent child-section) + home "Today's schedule" card |
| Web | Admin console: bell schedule & period CRUD, section grid editor (drag-free, conflict warnings), teacher view, CSV export, year/class/section/teacher filters |

- ✅ 3 — Relationships + **delete-rule rollback probes** (one rolled-back tx, 0 rows persisted): AcademicYear /
  Section / Subject / Period / BellSchedule / User(teacher) all **Restrict-blocked**; `TeacherAssignment`

## Migration proof (additive, zero frozen-table change)

`prisma migrate diff` (baseline = all M1–M8 migrations → new schema) emitted **only**:
`CREATE TYPE "Weekday"`, `CREATE TABLE "BellSchedule"|"Period"|"TimetableEntry"`, their indexes/uniques,
7× `ADD FOREIGN KEY … ON DELETE RESTRICT`, and the `Period` CHECK — **no `ALTER` on any frozen table**.
Applied to a fresh `m9_verify` DB atop all prior migrations; `migrate diff` DB→schema = **No difference**
(zero drift). The four back-relations on AcademicYear/Section/Subject/User are virtual (emit no SQL).

## RLS proof (isolation, rolled back)

| Actor | TimetableEntry visible | Result |
|---|---|---|
| Teacher A | own rows only | ✓ cannot read Teacher B |
| Teacher B | own rows only | ✓ (A ≠ B) |
| Parent (child in sec A) | sec A only | ✓ cannot read sec B |
| Admin | all | ✓ |
| Anon | none | ✓ |
| Reference (BellSchedule/Period) | teacher+parent read; anon none | ✓ |

## Conflict proof (business tests)

Section overlap · teacher overlap · duplicate period → `ConflictError`; ownership missing → `ValidationError`
(never ClassTeacherAssignment); break period / cross-year → `ValidationError`; update excludes self; period
time overlap / order clash / `start≥end` / `order≤0`; one bell schedule per year.

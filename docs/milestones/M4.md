# M4 — Attendance Management — ✅ STEPS 1–10 COMPLETE (awaiting approval)

Scope (per M4 kickoff brief): **Attendance (Session + Record), Leave Request,
Attendance Correction, Holiday, and a compute-on-read Attendance Summary.** Out:
homework, exams, marks, report cards, fees, timetable, notifications — later
milestones.

Roles: SUPER_ADMIN + OFFICE_ADMIN full management; TEACHER marks + reads own
sections and submits corrections; PARENT reads own children, applies for leave.
ACCOUNTANT none. **ADR-011 is the source of truth**: attendance keys to
**Enrollment, never Student** (history survives promotion); a register is an
`AttendanceSession` holding `AttendanceRecord`s; ownership derives from
`TeacherAssignment` (never stored); lifecycle is forward-only `DRAFT → SUBMITTED
→ LOCKED`; approved leave biases the marking default without eager writes;
corrections are immutable and update the record only on approval.

## Steps

- ✅ 1 — Requirements analysis: daily/subject/leave/correction workflows, teacher
  ownership, parent visibility, holiday handling, half-day/late rules, report-card
  dependency; why attendance belongs to Enrollment and why history survives
  promotion. Recorded in **ADR-011** + PERMISSIONS_MATRIX update. Refinements
  folded in: session audit actors (created/submitted/locked, not an owner),
  immutable corrections (previous/requestedStatus), leave-as-default-only,
  `DRAFT→SUBMITTED→LOCKED`, working-day **calendar** (not a service override flag).
- ✅ 2 — Database design (migration `20260707000000_attendance_management`): 5
  models (`AttendanceSession`, `AttendanceRecord`, `LeaveRequest`,
  `AttendanceCorrection`, `Holiday`) + 6 enums. Invariants in the DB: two partial
  unique indexes for the register natural key (DAILY null / SUBJECT not-null),
  `(sessionId, enrollmentId)` unique, `(academicYearId, date)` holiday unique,
  `fromDate ≤ toDate` CHECK. Empirically verified against Postgres (drift-check +
  6 seeded constraint proofs). Additive `submittedAt/lockedAt` + `remarks`.
- ✅ 3 — Relationships: `AttendanceRecord` **Cascade** from session, **Restrict**
  to Enrollment (attendance blocks hard-delete, ADR-011 §1); session actor FKs +
  leave/correction actor FKs **Restrict** (audit actors durable); correction
  **Cascade** from record; `Holiday` **Cascade** from AcademicYear (year-owned
  config). All 7 delete-rule edges proven empirically.
- ✅ 4 — RLS (migration `20260707010000_attendance_rls`): admin ALL; teacher
  own-assigned-section SELECT (session/record/leave/correction); parent own-child
  SELECT (record/leave); calendar readable by any authenticated user; anon none.
  Isolation proven with a stubbed `auth.uid()` across two families/sections.
- ✅ 5 — Business layer: 4 services (`attendance`, `leave`, `correction`,
  `holiday`) + shared `scope.ts` (staff-actor B3 resolution, teacher/parent/
  working-day scope, name resolver) + mappers. Permission → scope → rule checks,
  typed DomainErrors, every mutation audited in-tx. Leave biases the roster
  default only; summary is compute-on-read (ADR §10 weighting).
- ✅ 6 — API layer: 4 thin routers (`attendance/leave/attendanceCorrection/
  holiday`) on `protectedProcedure`; Zod inputs in `@repo/validation` (dates
  transform at the boundary, the codebase convention). No logic/Prisma/role
  strings in transport.
- ✅ 7 — Mobile: teacher — mark (find-or-open → roster → save/submit/lock),
  history, my-corrections + submit-correction; parent — child attendance +
  calendar (list), apply leave + status. No admin CRUD. Two additive reads
  surfaced: `attendance.findSession`, `attendanceCorrection.listMine`; and
  `AttendanceRecordDto` gained `date` (for history/calendar).
- ✅ 8 — Web: `/attendance/*` dashboard — mark (bulk entry, filters, date picker,
  CSV export, submit/lock), summary (per-student %, CSV), leave + correction
  **approval queues** (enriched with student names), holiday management. Two
  additive admin reads: `leave.listPending`, enriched `attendanceCorrection.
  listPending`.
- ✅ 9 — Testing: business (`attendance.services.test.ts` +
  `attendance.concurrency.test.ts`) — workflows, full edge-case bank, a
  parameterized **state-machine matrix**, and **real `Promise.all` concurrency**
  races; API (`attendance.test.ts`) — transport gates, Zod, a parameterized
  **authorization matrix** (every procedure × every role); validation edges.
  **Concurrency hardening (real defect fixed):** session transitions + correction
  approval now use guarded conditional updates (`updateMany WHERE status=<from>`)
  → exactly one transition + one audit, loser gets `Conflict`.
- ✅ 10 — Documentation: API_INVENTORY attendance section (implemented) +
  re-tagged the draft holiday row; `features/attendance.md`; `status/Attendance.md`;
  `project_memory` + this file. **No new ADR** — ADR-011 (Step 1) covers the
  decisions; the concurrency guard is an implementation detail of it.
